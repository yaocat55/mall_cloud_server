# Mall Cloud 架构改进建议

基于 `mall_cloud_server/` 项目结构通读，以下是按优先级排序的改进建议。

---

## P0 - 安全 & 配置管理

### 1. 敏感信息外部化（已完成）

> ✅ 已创建 `application.yml.template` + `.gitignore` 过滤 `application.yml`

所有密码、密钥、IP 仍硬编码在 `.yml` 文件中。后续建议进一步：

- **改用环境变量注入**：`password: ${MYSQL_PWD:your_mysql_password}`，默认值 fallback 到模板值
- **或集成配置中心**：如 Nacos/Apollo/Spring Cloud Config，完全不在本地文件存储凭据

### 2. JWT 鉴权：内部服务不再依赖 Redis（已实施）

**现状（已改造完成）**：

```
Gateway AuthFilter:
  ① JWT 验签（签名 + 过期）
  ② 透传 X-User-Id / X-User-Name / X-Roles Header
  不碰 Redis

下游服务 AuthApiInterceptor:
  ① 优先从 Gateway Header 取 userId / username / roles
  ② 设入 SecurityContext
  0 次 Redis

mall-auth JwtTokenFilter（唯一碰 Redis 的地方）:
  ① 解析 JWT claims
  ② Redis 查 blacklist:{jti}（仅 admin 踢人场景）
  ③ 设入 SecurityContext
```

**关键变化**：
- Redis 从鉴权链路中剥离，不再在 Gateway 或下游服务中使用
- Redis 退化 mall-auth 内部的黑名单存储，仅用于：
  - **黑名单**：`blacklist:{jti}` 存储被踢 token 的 jti
  - **用户 jti 映射**：`user_token_jti:{userId}` 存当前 jti，踢人时查
- C 端消费者请求全程不碰 Redis
- Gateway 启动不再依赖 Redis

### 3. 移除 `.idea/` 中的敏感信息

`.idea/` 目录已加入 `.gitignore`，但建议确认 IDE 工作区文件（`workspace.xml`、`compiler.xml`）未包含数据库密码或服务器 IP。

### 4. 库存模型是玩具级的三段式

**现状**：
- `ProductEntity` 只有一个 `stock` 字段，无 `lockStock`
- 下单流程：`reduceStockBatch` → 直接扣 `stock`。订单创建失败 → MQ 补偿回滚
- 超时取消订单 → 不恢复库存

**问题**：
- `stock` 字段既表示"可卖数"又表示"物理库存"，语义混乱
- 用户下单不支付，库存被扣死
- MQ 回滚不可靠（生产者发送失败 / RocketMQ 5.x 兼容 / 无兜底定时任务）
- 超时取消不恢复库存（`cancelTimeoutById` 只改订单状态）
- 无 Redis 层，高并发下单直接打 MySQL，扛不住秒杀

**建议：三层降级 + 分段语义**

```
请求链路：
           +-----------+   命中?    +--------+   命中?    +---------+
请求 →     | 本地缓存  | ──否──→   | Redis  | ──否──→   | MySQL   |
(扣库存)    | Caffeine  |          |        |           | (兜底)   |
           +-----------+          +--------+           +---------+
               200ms TTL           RDB+AOF持久化         最终一致
```

**第一层 — 本地缓存（Caffeine，JVM 级）：**

```java
// 秒杀/热卖商品库存放本地缓存
Cache<String, Integer> localStock = Caffeine.newBuilder()
    .expireAfterWrite(200, TimeUnit.MILLISECONDS)   // 200ms 过期
    .maximumSize(10000)
    .build();
```

- 仅缓存热卖商品（最近 5 分钟下单量 TOP 100）
- 200ms 自动失效，回源 Redis
- 扛掉 99% 的热点请求

**第二层 — Redis（主战场）：**

```
Key: "stock:pid:{productId}"                ← 单键
Key: "stock:pid:{productId}:{shardIndex}"   ← 热商品分片键

下单流程：
  ① DECR stock:pid:{id}          ← Redis 原子扣减，~1ms
  ② 结果 < 0 → INCR 补回 + 抛异常
  ③ >= 0 → 继续下单流程

支付回调：
  ① 不下发 Redis，交由定时任务同步 MySQL

超时/取消：
  ① INCR stock:pid:{id}          ← Redis 加回
```

**第三层 — MySQL（持久化兜底）：**

```sql
-- ProductEntity 只保留物理库存，可卖数走 Redis
ALTER TABLE `product_product` ADD COLUMN `sold_quantity` int DEFAULT 0 COMMENT '已售数量';
-- stock 语义：物理库存（总可卖数），不做实时扣减
```

```
定时任务（每分钟）：
  ① 扫描 Redis 中库存 Key
  ② 找出已售罄的（stock = 0），清理 Redis Key
  ③ 批量回写 MySQL：
     UPDATE product_product 
     SET sold_quantity = ?, 
         stock = stock - ?
     WHERE id = ?
  
  ④ 对账：Redis 当前库存 + MySQL 已售  是否 ≈ MySQL 初始库存
      发现不一致 → 告警 + 以 Redis 为准修正 MySQL
```

```sql
-- 定时任务记录库存流水，方便对账排错
CREATE TABLE `product_stock_log` (
    `id`          bigint       NOT NULL,
    `product_id`  bigint       NOT NULL,
    `change_type` varchar(32)  NOT NULL COMMENT 'order:下单 cancel:取消 pay:支付 refund:退款',
    `quantity`    int          NOT NULL COMMENT '变动数量（负数=扣减）',
    `redis_stock` int          NOT NULL COMMENT '变动后 Redis 库存',
    `order_id`    bigint                COMMENT '关联订单',
    -- BaseEntity
    PRIMARY KEY (`id`),
    KEY `idx_product_id` (`product_id`)
) COMMENT '库存变动流水表';
```

**热商品分片键：**

对于单商品秒杀 QPS > 10000 的场景，一个 Redis Key 扛不住：

```
Key 设计：stock:pid:{id}:{shard}
分片数：1024
扣减：  随机选一个 shard 做 DECR

拼回时 SUM 所有 shard
```

业务初期不需要，但架构上留好接口：

```java
interface StockService {
    boolean deduct(Long productId, int quantity);           // 普通商品
    boolean deductShard(Long productId, int shard, int quantity); // 热商品分片
    void syncToMysql(Long productId);                       // 回写 MySQL
}
```

**三段式语义（通过订单状态驱动 Redis）：**

```
下单   → Redis DECR（预占库存）         ← 锁定
支付   → 不操作 Redis（等定时任务同步） ← 真卖
超时   → Redis INCR（释放预占库存）     ← 归还
取消失败 → Redis INCR                  ← 归还
```

**完整降级流程：**

```
正常：本地缓存 → Redis → MySQL 兜底同步
Redis 宕机：降级到 MySQL 直接扣（降级）
MySQL 宕机：Redis 照常扣，恢复后同步
本地缓存穿透：回源 Redis，不影响 MySQL
```

这样做的好处是日常 99% 的库存请求止步于 Redis，MySQL 只做持久化和对账，不参与实时扣减。热商品还能用分片键解决单 Key 热点问题。

### 5. mall-pay 无数据库：支付记录缺失

**现状**：`mall-pay` 排除了 `DataSourceAutoConfiguration`，无数据库、无 mapper、无支付记录表。每次支付成功后直接 Feign 调用 mall-order 更新订单状态，支付服务自身不保留任何记录。

**影响**：
- 无法做支付对账（支付宝账单 vs 本地记录）
- 退款时查不到原始支付记录，需依赖订单服务的数据
- 审计和排查问题缺少依据

**建议**：新增支付订单表 `pay_order`，记录支付流水号、金额、状态、支付宝回执等字段，支持对账和退款溯源。

### 6. 商品快照缺失

**现状**：`OrderService` 创建订单时，订单明细（`t_order_item`）只存了 `product_id`、`quantity`、`price` 三个字段。商品名称、图片、规格等字段依赖实时关联商品服务。

**问题**：
- 三个月后用户看历史订单，如果商品已下架或改名，订单页显示异常
- 商家修改商品图片后，历史订单的图片跟着变——订单是财务凭证，不应该依赖实时数据
- 退款/售后时需要展示"下单时的商品信息"，而不是当前信息

**建议**：

```sql
-- t_order_item 增加快照字段
ALTER TABLE `t_order_item` ADD COLUMN `product_snapshot` json COMMENT '下单时商品快照';
-- 快照内容：
-- { "name": "iPhone 15", "image": "url", "spec": "256G 黑色", "brand": "Apple" }
```

在下单时，从商品服务查出完整商品信息，连同 ID 一起写入订单明细。后续任何时候查看历史订单，直接读快照字段，不调商品服务。

### 7. 缺少统一价格计算引擎

**现状**：价格计算散落在各处：
- 购物车 `ShoppingCartService` 算一次（`cart.setPrice(product.getPrice())`）
- 提交订单 `OrderService.java:440` 读购物车价格，不从商品服务重新获取
- 优惠券抵扣在 `mall-marketing` 算
- 会员折扣无处安放
- 满减活动如果要加，又得改一处

**问题**：
- 加一个新优惠类型就要改 N 个文件
- 购物车价格和下单结算价可能不一致（加购期间商家改价）
- 无法对最终价格做完整审计

**建议**：定义统一价格计算管道：

```
商品基础价 → 会员折扣价 → 活动优惠价 → 优惠券抵扣 → 运费 → 实付
```

每个环节是一个独立的 `PriceCalculator` 接口，用责任链模式串联。新增优惠类型只需加一个实现类，不影响已有逻辑。

### 8. 全链路缺乏幂等防护

**现状**：目前仅在 `CouponService.receiveCoupon()` 提到幂等（#10），但问题远不止优惠券：
- **下单提交**：用户快速双击 → 生成两笔订单，扣两次库存
- **支付回调**：支付宝/微信会重复通知，`PayService.notify()` 没有幂等
- **退款申请**：用户多次点击 → 生成多个退款单
- **优惠券领取**（已有记录）

**建议**：在 `mall-common` 中统一提供幂等能力：

```java
// 1. 幂等注解
@Idempotent(key = "#userId + ':' + #scene", timeout = 5)
public Result submitOrder(Long userId, OrderSubmitDTO dto) { ... }

// 2. 幂等工具类（Redis SETNX 实现）
IdempotentUtil.execute("order:submit:" + userId, () -> submitOrder(userId, dto), 5, TimeUnit.SECONDS);
```

幂等 key 的设计原则：`业务域:操作:用户ID:业务唯一标识`
所有写接口必须做幂等，支付回调这类危险接口优先接入。

### 9. 退款状态机不完整

**现状**：`OrderReturnApplyService` 中仅有 `applyStatus` 字段，缺少 `WAIT_REFUND → REFUNDING → REFUNDED → CLOSED` 的标准退款状态机和流转校验。

**建议**：定义退款状态枚举，约束状态合法流转路径。

### 10. 接入测试体系

**现状**：`maven-surefire-plugin` 配置 `<skip>true</skip>`，整个项目**零测试**，`src/test/java` 目录不存在。

**建议**：
- 逐步接入分层测试：
  - **单元测试**：JUnit 5 + Mockito，覆盖工具类、Service 层核心逻辑
  - **集成测试**：Testcontainers（MySQL、Redis、ES）验证数据访问层
  - **契约测试**：验证 Feign Client 接口签名一致性
- 去除 `<skip>true</skip>`，把测试纳入构建流水线

### 11. 缺少统一异常处理

**现状**：各服务没有 `GlobalExceptionHandler` 或 `RestControllerAdvice`。异常处理散落在各 Controller 中，返回格式不一致，前端难以统一处理错误。

**建议**：在 `mall-common` 中定义统一的 `GlobalExceptionHandler`，统一错误响应格式。

### 12. BFF 服务缺少 scanBasePackages

**现状**：`mall-admin-api` 和 `mall-mobile-api` 的 `@SpringBootApplication` 未设置 `scanBasePackages`，默认扫描整个包树，可能加载不需要的 Bean。

对比其他服务均已显式声明：
```java
@SpringBootApplication(scanBasePackages = {"cn.net.mall.auth"})
```

**建议**：为两个 BFF 服务补上 `scanBasePackages`，遵循已有模式。

### 13. 缺失 application.yml.template

**现状**：`mall-admin-api` 和 `mall-mobile-api` 没有 `application.yml.template`，新成员部署时不知道需要配什么。

其余 8 个服务都有。

### 14. Feign 缺少超时/熔断配置

**现状**：所有服务的 Feign 调用没有配置 connect-timeout、read-timeout、retry 或 Sentinel 熔断。任何一个服务慢或挂掉，调用方会一直等待默认超时（通常 60s+），导致级联故障。

**建议**：统一配置 Feign 超时 + Sentinel 熔断降级规则。

### 15. 引入 Docker 本地开发环境

**建议**：在项目根目录添加 `docker-compose.yml`，一键启动所有中间件，参考 README 中的基础设施清单。

### 16. 搭建 CI/CD 流水线

**现状**：零 CI/CD 配置。

## P2 - 代码 & 架构

### 1. 移动端用户与管理员用户表分离

**现状**：小程序用户和管理后台用户共用 `auth_user` 表，共用同一个 `UserService` 和 `UserMapper`。所有操作都落在 `mall_auth` 库的 `auth_user` 表中。

**核心问题：两个完全不同的权限模型硬塞在一张表里**

| 维度 | Admin 管理后台 | 移动端（小程序） |
|------|---------------|-----------------|
| **权限模型** | RBAC（角色 → 菜单 → 权限） | 会员权益体系 |
| **认证方式** | 用户名 + 密码（RSA解密+BCrypt） | 手机号 + 短信验证码 |
| **核心字段** | `dept_id`、`job_id`、`role` 关联 | 无对应字段，全为 null |
| **需扩展字段** | — | 会员等级、积分、成长值、优惠券额度 |
| **用户来源** | 管理员手动创建 | 用户自主注册 |

**具体冲突**：

1. **字段互斥**：`dept_id`、`job_id`、`role` 对移动端用户永远是 null；而移动端需要的 `vip_level`、`points`、`growth_value`、`total_consumption` 等字段 `auth_user` 上根本没有，也无法加 —— 加了管理端用户也用不到。

2. **认证逻辑打架**：
   - Admin 登录走 `UserDetailsServiceImpl.loadUserByUsername()` → 查 `password` 字段做 BCrypt 校验
   - 移动端注册/登录走 `SmsAuthenticationProvider` → 手机号 + 短信码，无密码
   - 同一个 `UserService` 里混着两套认证逻辑，`register()` 方法里既有图形验证码校验又有短信验证码校验

3. **手机号冲突**：一个手机号如果在移动端注册了，Admin 端就不能再用这个手机号创建管理员账号 → `findByPhone` 已经查到人了。反过来也成立。

4. **用户列表污染**：Admin 端的用户管理页面会列出所有移动端注册用户，`searchByPage` 拿到的数据里混着大量无部门/无角色的普通用户。

**建议**：
- **新建 `member`（或 `ums_user`）表**存储移动端用户，`auth_user` 改为纯管理员表
- **会员表独立字段**：`vip_level`、`points`、`growth_value`、`total_consumption`、`register_source`（微信/支付宝/手机号）
- **认证链路拆分**：
  - Admin 端保留用户名+密码登录，走 `UserDetailsService`
  - 移动端完全走手机号+短信验证码，互不干扰
- **登录接口校验用户类型**：Admin 登录接口只查 `auth_user`，移动端接口只查 `member` 表
- **两个 BFF 各管各的**：`mall-admin-api` 的登录注册对接 `auth_user`，`mall-mobile-api` 对接 `member` 表
- **历史数据迁移脚本**：将现有 `auth_user` 中 dept_id/job_id 全为 null 的用户迁移到 `member` 表

### 2. 认证模块功能不完整

`mall-auth` 仅实现了登录注册和 RBAC 基础模型，缺乏：

- 用户状态管理（禁用/冻结/锁定）
- 多设备登录策略（单设备/多设备限制）
- 登录安全（失败锁定、异地检测）
- OAuth2 / 三方登录

### 3. product_favorites 同步机制不明确

`product_favorites` 数据分别存储在 mall-product 的单库和 mall-recommend 的分库分表中，两个库之间的同步路径不存在。

**建议**：明确同步路径（RocketMQ 或定时批处理），定义最小消息契约替代共享表结构。

### 4. 大量 DTO 缺少 @Schema 注解

**现状**：`mall-product-client` 中 15 个 DTO 没有 `@Schema(description = "...")` 注解（包括 `ProductDetailInfoDTO`、`ShoppingCartBuyDTO` 等核心 DTO），Swagger 文档中显示为无描述字段。

**建议**：逐批补齐。

### 5. 缺失 Feign 超时/熔断配置

同 P1 #4。

### 6. 缺失 client 模块

**现状**：`mall-message` 和 `mall-recommend` 没有对应的 `*-client` 模块，其他服务如果需要调用它们，只能直接依赖完整的服务模块而非轻量 client。

**建议**：抽取 `mall-message-client` 和 `mall-recommend-client`，遵循已有模式。

### 7. 统一错误码体系

**现状**：异常处理分散在各服务，`BusinessException` 使用 `ASSERT_ERROR_CODE` 这类常量，缺少中心化错误码定义。

**建议**：在 `mall-common` 中定义枚举。

### 8. 事务注解使用不一致

**现状**：部分 Service 方法标注了 `@Transactional(rollbackFor = Throwable.class)`，部分没有。`mall-product` 使用 `transactionTemplate.execute` 编程式事务而非声明式事务。缺少统一的事务管理规范。

### 9. 统一日志 & 链路追踪

**现状**：`logback-spring.xml` 各服务各有一份，缺链路追踪基础设施。

**建议**：
- 在 `mall-common` 中定义统一 `logback-spring.xml` base，各服务继承
- 集成 SkyWalking 或 Micrometer Tracing

## P3 - 开发体验

### 1. 缺少 application.yml.template（BFF）

同 P1 #3。

### 2. 添加 Maven Wrapper

根目录缺 `mvnw` / `mvnw.cmd`，新成员需要自行安装 Maven。

### 3. 启动顺序自动化

**现状**：需要手动按阶段顺序启动 9 个服务。

**建议**：提供一个 `start.sh` / `start.bat` 脚本，自动按依赖顺序启动各服务。

## 总结

| 优先级 | 类别 | 核心行动 |
|--------|------|---------|
| P0 | 库存 | 三段式改造、移除 MQ 回滚依赖 |
| P0 | 安全 | JWT claims 透传、全链路幂等 |
| P0 | 业务 | 支付入库、商品快照、价格引擎 |
| P1 | 工程基建 | 测试、异常处理、scanBasePackages、template、Feign 超时 |
| P2 | 代码/架构 | 用户表分离、DTO @Schema、缺失 client、事务规范 |
| P3 | 开发体验 | Maven Wrapper、启动脚本 |
