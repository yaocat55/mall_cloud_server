# 分布式交易与配额处理 —— 技术要点

本文聚焦系统中最核心的技术难点：**分布式事务一致性**、**高并发额度扣减**、**交易状态机与延迟调度**。剥离具体业务属性，提炼通用的分布式系统设计思路。

> **适用场景：** 无论是电商下单扣库存、积分商城兑换扣额度、API 网关扣调用次数、活动报名锁名额、还是优惠券发放控制余量，底层技术模型完全一致——都是"预占额度 → 确认扣减 → 超时释放"三段式 + 状态机 + 最终一致性。

---

## 一、交易状态机与延迟调度

### 1.1 状态机设计

交易是全系统最核心的数据实体，状态流转要求严格的前后置校验。

```
状态图：

                    ┌── 超时释放 ──→ CANCEL
                    │
WAIT_CONFIRM ── 确认 ──┼── 手动取消 ──→ CANCEL
    │                  │
    │                  └── 确认成功 ──→ CONFIRMED ─── 发放 ──→ DELIVERED ────→ COMPLETED
    │                                                                           │
    │                                                                      申请回退
    │                                                                           │
    │                                                                      REFUNDING
    │                                                                           │
    │                                                                       REFUNDED
    │
    └── 超时未确认 ──→ CANCEL（释放预占额度）
```

> **场景映射：** 电商的"支付"对应交易的"确认"，"发货"对应"资产发放"（积分入账、API 密钥生效、活动资格生效）。状态机本身与业务无关。

**技术实现关键点：**
- 状态枚举定义合法的 **from → to** 转换集合
- 每个状态的进入和退出都有 side effect（如 CANCEL 触发额度释放）
- 不允许跳状态（如直接从 WAIT_CONFIRM 到 COMPLETED）

### 1.2 延迟消息驱动超时释放

交易超过 30 分钟未确认需要自动释放。这里面临的问题是：**千万级交易流水，怎么高效地做超时检测？**

**业界方案对比：**

| 方案 | 实现 | 问题 |
|------|------|------|
| 定时任务扫全表 | `SELECT * FROM trade WHERE status=WAIT_CONFIRM AND create_time < now()-30min` | 数据量大时慢，每次扫全表 |
| 定时任务扫时间分区 | 按创建时间分区扫描 | 复杂，维护成本高 |
| Redis keyspace notification | 设置 Key 过期事件 | 不可靠，可能丢消息 |
| **RocketMQ 延迟消息** | 发起交易时发一条延迟 30 分钟的 MQ 消息，消费时检查状态 | **异步解耦，精准触发** |

**项目采用 RocketMQ 延迟消息的方案：**

```java
// 发起交易成功后
mqHelper.sendDelay(
    TRADE_TIMEOUT_CANCEL_TOPIC,   // Topic
    delayMsg,                     // 交易ID
    15                            // 延迟等级 15 = 30分钟
);
```

```java
// 消费者：到期后检查状态，如果还是 WAIT_CONFIRM 则释放
@RocketMQMessageListener(
    topic = "TRADE_TIMEOUT_CANCEL_TOPIC",
    consumerGroup = "trade_timeout_cancel_group"
)
public void onMessage(MessageExt message) {
    TradeEntity trade = tradeService.findById(tradeId);
    if (trade.getStatus() == TradeStatus.WAIT_CONFIRM) {
        tradeService.cancel(tradeId);  // 取消 + 释放额度
    }
}
```

**注意点：**
- RocketMQ 延迟等级是预定义的（level 1-18），不是任意秒数
- 消费端要做幂等：消息可能重复投递
- 兜底：定时任务扫描已超时但未被释放的交易（消息丢失防护）

### 1.3 为什么选 MQ 延迟消息而不是定时任务？

```
定时任务扫全表：
  交易量 100 万 → 每次扫描 100 万条 → 每分钟一次 → 数据库压力大
  大部分交易早就确认了，不需要处理

MQ 延迟消息：
  每个待确认交易对应一条消息 → 到期后精准触发 → 0 空扫描
  即使 100 万交易，每分钟最多处理"刚好过 30 分钟"的那一批 ≈ 几百条
```

这是典型的 **"拉模式转推模式"** 的思路。

## 二、高并发额度扣减

### 2.1 问题本质

额度扣减是典型的高并发写场景，核心矛盾是：

> **扣减操作必须原子性 + 不超发，但数据库行锁在高并发下是瓶颈。**

所有方案都是在这个矛盾上做权衡。

### 2.2 三层降级架构

```
请求链路：
           +-----------+   未命中   +--------+   未命中   +--------+
请求 →     | 本地缓存  | ───────→  | Redis  | ───────→  | MySQL  |
(扣额度)    | Caffeine  |           | DECR   |           | 兜底   |
           | 200ms TTL |           | 1-5ms  |           | 定时   |
           +-----------+           +--------+           +--------+

降级策略：
  Redis 不可用 → MySQL 直扣（性能下降，功能正常）
  本地缓存穿透 → 回源 Redis（正常路径）
  MySQL 不可用 → Redis 继续扣，恢复后同步（最终一致）
```

> **场景映射：**"额度"可以是库存数量、API 调用次数、优惠券余量、活动名额、虚拟币余额——只要是"总数有限、先到先得"的模型，架构不变。

### 2.3 Redis 原子扣减

```java
// Redis 扣减（原子操作）
Long result = redisUtil.decr("stock:pid:" + productId);
if (result < 0) {
    redisUtil.incr("stock:pid:" + productId);  // 补回
    throw new BusinessException("库存不足");
}
```

**为什么用 DECR 而不是 GET + 比较 + SET？**
- DECR/INCR 是 Redis 原子命令，不需要 Lua
- GET + 比较 + SET 不是原子的，高并发下条件竞争 → 超发
- Lua 脚本也可以，但 DECR 一行搞定

### 2.4 热点资源分片 Key

单个 Redis Key 最多扛 10 万 QPS（单线程模型），遇到秒杀级别的热点，需要拆分：

```java
// 热点分片（1024 片）
String buildShardKey(Long resourceId) {
    int shard = ThreadLocalRandom.current().nextInt(1024);
    return "quota:" + resourceId + ":" + shard;
}

// 扣减：随机选分片
Long result = redisUtil.decr(buildShardKey(resourceId));

// 查询总余量：SUM 所有分片
// 实际生产中不需要实时 SUM，定时任务回写 MySQL 时顺便 SUM
```

**分片 Key 的粒度选择：**
```
1024 片 → 单 Key 压力 = 总 QPS / 1024，分布式均匀
          Redis Cluster 下还能分布到不同节点
          初始化时预创建，不动态增加
```

### 2.5 本地缓存挡热点

对于 200ms TTL 的本地缓存，要考虑的：

```
为什么 200ms？
  TTL 太长 → 余量更新不及时 → 可能超发
  TTL 太短 → 每次都穿透到 Redis → 缓存意义不大
  200ms 在"及时性"和"命中率"之间取平衡

谁来缓存？
  只缓存热点资源——最近 5 分钟请求量 TOP 100
  冷门资源不需要缓存，直接走 Redis
```

### 2.6 定时兜底：最终一致

MySQL 不做实时扣减，只做最终一致性持久化：

```java
@Scheduled(fixedDelay = 60_000)  // 每分钟执行
public void syncQuotaToMysql() {
    // 1. 扫描所有 Redis 额度 Key
    Set<String> keys = redisUtil.scan("quota:*");

    for (String key : keys) {
        Long resourceId = extractResourceId(key);
        int redisQuota = redisUtil.get(key);

        // 2. 回写 MySQL
        quotaMapper.updateQuota(resourceId, redisQuota);

        // 3. 如果已耗尽（quota = 0），清理 Redis Key
        if (redisQuota <= 0) {
            redisUtil.del(key);
        }
    }

    // 4. 对账
    long mysqlTotal = quotaMapper.sumConsumed();
    long redisTotal = sumAllRedisKeys(keys);
    if (Math.abs(mysqlTotal - redisTotal) > threshold) {
        alertService.send("余量不一致，差 " + (mysqlTotal - redisTotal));
    }
}
```

### 2.7 额度变动流水追溯

```sql
CREATE TABLE `quota_change_log` (
    `id`            bigint       NOT NULL,
    `resource_id`   bigint       NOT NULL COMMENT '资源ID（商品/活动/API Key）',
    `change_type`   varchar(32)  NOT NULL COMMENT 'deduct:扣减 refund:回退 admin:调整',
    `quantity`      int          NOT NULL COMMENT '变动数量（负数=扣减）',
    `redis_quota`   int          NOT NULL COMMENT '变动后 Redis 余量',
    `trade_id`      bigint                COMMENT '关联交易ID',
    `create_time`   datetime     NOT NULL,
    PRIMARY KEY (`id`),
    KEY `idx_resource_id` (`resource_id`)
) COMMENT '额度变动流水';
```

**为什么需要流水表？**
- 对账：发现不一致时，可以查流水定位
- 审计：运营想知道某个资源为什么突然超发了
- 回放：系统崩溃恢复后，可以重新回放流水重建 Redis 缓存

## 三、分布式事务一致性

### 3.1 跨服务事务问题

交易创建涉及的服务调用链：

```
TradeService.create()
  → ResourceService: 扣减额度（远程调用）
  → PromotionService: 核销优惠/营销（远程调用）
  → TradeService: 保存交易记录（本地事务）
```

这三个操作不能放在同一个数据库事务里（跨服务跨数据库）。任何一个步骤失败，都需要补偿。

### 3.2 补偿方案演进

**方案一：MQ 回滚补偿（当前实现）**

```
发起交易 → 扣额度 → 保存交易 → 成功 ✓
                     ↓ 失败
               发 MQ 回滚额度
```

**问题：** MQ 可能发送失败，Producer 挂了消息就丢了，额度永久不回滚。

**方案二：Redis 预扣（推荐方案）**

```
发起交易 → Redis DECR 扣额度 → 保存交易 → 成功 ✓
                         ↓ 失败
                   Redis INCR 释放（本地 catch）
```

**关键区别：** Redis 的 INCR 在交易创建失败时直接在本服务的 catch 块中执行，不需要跨网络发 MQ。单次操作原子性由 Redis 保证，回滚由本地异常处理保证。

**方案三：定时任务兜底（最终防线）**

```
定时扫描：所有 Redis 中被扣减
        但超过 N 分钟没有对应有效交易的记录
        自动 INCR 释放
```

### 3.3 幂等防护（防止重复操作）

幂等是分布式系统中最容易被忽视的问题。写接口不做幂等，遇到网络重试、用户双击、MQ 重投就会出数据错误。

```java
@Idempotent(key = "#userId + ':' + #tradeId", timeout = 5)
public void submitTrade(Long userId, Long tradeId) { ... }
```

```
幂等 Key 设计原则：业务域:操作:用户ID:业务唯一标识

例子：
  trade:submit:123:20230701001        ← 发起交易
  pay:notify:123:pay_trade_no         ← 支付回调
  coupon:receive:123:S20230701        ← 领取优惠
```

**Redis SETNX 实现：**

```java
public boolean execute(String idempotentKey, Runnable action, int timeoutSeconds) {
    String key = "idempotent:" + idempotentKey;
    Boolean success = redisUtil.setIfAbsent(key, "1", timeoutSeconds, TimeUnit.SECONDS);
    if (Boolean.TRUE.equals(success)) {
        action.run();
        return true;
    }
    return false;  // 重复请求
}
```

## 四、数据分片

项目使用了 ShardingSphere 做分库分表，核心设计思路：

### 4.1 交易分片

```
8 个数据库（ds_trade_0 ~ ds_trade_7）
每个库 32 张表

分片键：trade_id（Snowflake 生成）
库路由：trade_id % 8
表路由：trade_id % 32

关联表（t_trade_item、t_trade_delivery）用 trade_id 绑定
→ 同一笔交易的所有数据在同一个库同一个绑定表组里
→ 关联查询不跨库
```

### 4.2 分片键选择

```
为什么用 trade_id 而不是 user_id？
  user_id 分片 → 一个用户的所有交易集中在同一分片 → 查询方便
  trade_id 分片 → 负载更均匀，但按用户查交易需要广播

权衡：
  项目选择 trade_id 分片，按用户查交易时走广播路由
  用户量初期不大，广播路由的损耗可接受
```

### 4.3 绑定表

同一笔交易相关的表在同一个分片中，避免跨库 JOIN：

```yaml
bindingTables:
  - t_trade, t_trade_item, t_trade_delivery
```

## 五、总结：技术亮点速查

| 领域 | 技术点 | 解决的问题 |
|------|--------|-----------|
| 延迟调度 | RocketMQ 延迟消息 | 替代定时任务扫全表，精准触发超时释放 |
| 状态机 | 状态枚举 + 前置校验 + side effect | 确保交易流转不越级、不丢失 |
| 额度扣减 | Redis DECR 原子操作 | 替代 MySQL 行锁，QPS 提升 100 倍 |
| 降级兜底 | Caffeine → Redis → MySQL | 三级兜底，任何一层挂了业务不中断 |
| 热点资源 | 1024 分片 Key | 单 Redis Key QPS 瓶颈扩展 |
| 最终一致 | 定时回写 + 流水对账 | Redis 数据不丢，系统可追溯 |
| 幂等 | Redis SETNX + 注解 | 网络重试/用户双击不产生重复交易 |
| 分片 | ShardingSphere trade_id % 8 | 单库写瓶颈扩展，绑定表不跨库 JOIN |
