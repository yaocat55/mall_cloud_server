# Admin 前端功能规划

## 核心理念

超级管理员能看到**全部菜单**，但唯一专属操作是**用户管理**（增删改查账号）。

角色、菜单、权限在架构设计阶段就已写死，不开放给前端配置：

```
用户创建 → 选部门（运营部/客服部/财务部）
         → 选岗位（主管/专员/...）
         → 权限自动关联（部门 → 角色 → 菜单 → 按钮权限）
```

## 超级管理员可见菜单

```text
系统管理（专属）
  └── 用户管理 → 创建/编辑/删除后台账号，选部门+岗位

商品管理（运营可见）
  全部 → 商品/分类/品牌/属性/单位/营销/首页

订单管理（客服可见）  
  全部 → 订单列表/退货审核

基础数据（运营可见）
  全部 → 图片/敏感词/短信/通知/评价
```

### 用户编辑字段

```
用户名（user_name）
手机号（phone）
邮箱（email）
部门（dept_id）        → 下拉选择：运营部 / 客服部 / 财务部
岗位（job_id）         → 下拉选择：主管 / 专员 / ...
状态（valid_status）   → 启用 / 禁用
密码（password）       → 重置密码
```

### 权限自动关联规则

| 部门 | 自动获得角色 | 可见菜单 |
|:----|:-----------|:---------|
| 运营部 | 运营 | 商品管理、营销管理、首页管理、基础数据、商品评价 |
| 客服部 | 客服 | 订单管理、商品评价 |
| 财务部 | 财务 | 订单管理（只读） |


## 不开发的前端页面

| 页面 | 原因 |
|:----|:------|
| 菜单管理 | 路由 + 权限标识由开发维护 |
| 角色管理 | 角色与部门的对应关系已写死 |
| 部门管理 | 部门为一次性配置，由开发维护 |
| 岗位管理 | 岗位为预设值，由开发维护 |
| 字典管理 | 技术常量，运营不需要理解 |
| 定时任务 | 技术运维 |

## 技术实现

- admin-bff 提供 `POST /admin/v1/user/page`（聚合：用户列表 + 部门列表 + 岗位列表）
- admin-bff 提供 `GET /admin/v1/user/{id}/edit-data`（聚合：用户详情 + 部门树 + 岗位列表）
- 后端根据用户的 `dept_id` 自动确定 `role_id`，写入 `auth_user_role`
- 菜单树由后端根据 `auth_role_menu` 查询返回，前端只负责渲染
