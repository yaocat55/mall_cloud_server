# Mall Cloud 架构改进建议

基于 `mall_cloud_server/` 项目结构通读，以下是按优先级排序的改进建议。

---

## P0 - 安全 & 配置管理

### 1. 敏感信息外部化（已完成）

> ✅ 已创建 `application.yml.template` + `.gitignore` 过滤 `application.yml`

所有密码、密钥、IP 仍硬编码在 `.yml` 文件中。后续建议进一步：

- **改用环境变量注入**：`password: ${MYSQL_PWD:your_mysql_password}`，默认值 fallback 到模板值
- **或集成配置中心**：如 Nacos/Apollo/Spring Cloud Config，完全不在本地文件存储凭据

### 2. JWT 鉴权：重复查 Redis

**现状**：Gateway 验签后全放行，每个下游服务的 `AuthApiInterceptor` 再次通过 Feign 或 Redis 查询用户信息恢复上下文。一次请求多一次网络开销。

**建议**：
- 将 `userId`、`roles`、`deptId` 等关键信息写入 JWT claims
- Gateway 完成验签后直接透传 claims 到下游服务
- 各服务本地解析 claims 恢复上下文，无需查 Redis
- Redis 仅用于黑名单（踢人下线）和 Token 刷新

### 3. 移除 `.idea/` 中的敏感信息

`.idea/` 目录已加入 `.gitignore`，但建议确认 IDE 工作区文件（`workspace.xml`、`compiler.xml`）未包含数据库密码或服务器 IP。

## P1 - 工程基建

### 3. 接入测试体系

**现状**：`maven-surefire-plugin` 配置 `<skip>true</skip>`，整个项目**零测试**，`src/test/java` 目录不存在。

**建议**：
- 逐步接入分层测试：
  - **单元测试**：JUnit 5 + Mockito，覆盖工具类、Service 层核心逻辑
  - **集成测试**：Testcontainers（MySQL、Redis、ES）验证数据访问层
  - **契约测试**：验证 Feign Client 接口签名一致性
- 去除 `<skip>true</skip>`，把测试纳入构建流水线

### 4. 引入 Docker 本地开发环境

**现状**：无 Dockerfile、无 docker-compose，新成员需要自行安装 MySQL/Redis/ES/RocketMQ 等 8 个中间件。

**建议**：在项目根目录添加 `docker-compose.yml`，一键启动所有中间件：

```yaml
version: '3.8'
services:
  mysql:      image: mysql:8.0   # ...
  redis:      image: redis:7     # ...
  nacos:      image: nacos/nacos-server:2.3.0
  elasticsearch: image: elasticsearch:7.17.21
  rocketmq:   image: apache/rocketmq:5.2.0
  mongodb:    image: mongo:7
  minio:      image: minio/minio
```

配合 `docker-compose up -d` + README 说明即可。

### 5. 搭建 CI/CD 流水线

**现状**：零 CI/CD 配置。

**建议**：添加 `Dockerfile` + `.github/workflows/ci.yml` 或 `Jenkinsfile`，实现：
- PR 时自动编译 + 跑测试
- 合并到主分支自动构建 Docker 镜像
- 可选：自动部署到测试环境

## P2 - 代码 & 架构

### 6. 移动端用户与管理员用户表分离

**现状**：小程序用户和管理后台用户共用 `auth_user` 表，共用同一个登录/注册接口。小程序注册用户也能登录管理后台，管理后台用户列表混入大量普通用户。

**建议**：
- 新建 `mobile_user` 表存储移动端用户，`auth_user` 改为纯管理员表
- 拆分 `MobileUserController` 和 `AdminUserController` 各自操作对应的表
- 登录接口校验用户类型，互不互通
- 两个 BFF 的登录注册接口分别对接各自的用户表
- 历史数据迁移脚本

### 7. 认证模块功能不完整

`mall-auth` 仅实现了登录注册和 RBAC 基础模型，缺乏：

- 用户状态管理（禁用/冻结/锁定）
- 多设备登录策略（单设备/多设备限制）
- 登录安全（失败锁定、异地检测）
- OAuth2 / 三方登录

### 8. product_favorites 同步机制不明确

`product_favorites` 数据分别存储在 mall-product 的单库和 mall-recommend 的分库分表中，两个库之间的同步路径不存在。

**建议**：明确同步路径（RocketMQ 或定时批处理），定义最小消息契约替代共享表结构。

### 9. 统一错误码体系

**现状**：异常处理分散在各服务，`BusinessException` 使用 `ASSERT_ERROR_CODE` 这类常量，缺少中心化错误码定义。

**建议**：在 `mall-common` 中定义枚举：

```java
public enum ErrorCode {
    SUCCESS(0, "成功"),
    UNAUTHORIZED(401, "未登录或 Token 已过期"),
    FORBIDDEN(403, "无权限"),
    NOT_FOUND(404, "资源不存在"),
    // 业务错误码范围
    USER_NOT_FOUND(1001, "用户不存在"),
    PRODUCT_STOCK_INSUFFICIENT(2001, "库存不足"),
    ORDER_NOT_FOUND(3001, "订单不存在"),
    // ...
}
```

配合 `GlobalExceptionHandler` 统一返回格式，减少 `e.getMessage()` 字符串传参。

### 7. 统一日志 & 链路追踪

**现状**：`logback-spring.xml` 各服务各有一份，缺链路追踪基础设施。

**建议**：
- 在 `mall-common` 中定义统一 `logback-spring.xml` base，各服务继承
- 集成 **SkyWalking**（README 中提到了依赖但未配置 agent）或 **Micrometer Tracing**（Spring Boot 3.x 原生），实现全链路 TraceId 透传

### 8. 对齐 pom.xml 元数据

**现状**：
- `mall-message`、`mall-order` 的 pom.xml 缺少 `<name>` 标签
- 各模块版本号未统一对齐（根 POM 版本 `1.0.0`，但无 `<dependencyManagement>` 统一管控三方库版本）

**建议**：补齐所有 `<name>`，并在根 POM 中用 `<dependencyManagement>` 锁定所有公共依赖版本。

### 9. 补齐缺失的 Feign Client

**现状**：大部分服务有 `*-client` 模块，但 `mall-message` 和 `mall-recommend` 没有，导致其他服务直接依赖完整的服务模块。

**建议**：为 `mall-message` 和 `mall-recommend` 抽取 `*-client` 模块，遵循已有模式。

### 10. 统一 SQL 文件位置

**现状**：建表 SQL 散落在各模块根目录 `sql/` 下（`mall-order/sql/`、`mall-product/sql/`、`mall-message/sql/`、`mall-recommend/sql/`）。

**建议**：统一放入模块内部的 `src/main/resources/sql/` 目录，随 JAR 一起打包，避免 `sql/` 目录与 `pom.xml` 同级带来的混乱。

### 11. 统一配置文件名规范

**现状**：
- 大部分服务用 `application.yml`
- `mall-order` 额外有 `bootstrap.yml`，但其他服务没有
- 配置中心 dataId 格式为 `mall-xxx-api-dev.yaml`，但本地文件名是 `application.yml`，两者不一致

**建议**：
- 统一使用 `bootstrap.yml`（仅含 Nacos 地址）+ `application.yml`（含所有其他配置）
- 或统一不使用 bootstrap，保持一致

### 12. 分库分表配置集中化

**现状**：仅 `mall-message` 有 `sharding.yaml`，`mall-order` 和 `mall-recommend` 也分表但未看到对应配置文件（可能通过 Nacos 远程管理）。

**建议**：
- 统一本地 ShardingSphere 配置文件路径（`src/main/resources/sharding/`）
- 或在 README 中注明哪些配置在 Nacos 中远程管理，避免新成员困惑

## P3 - 开发体验

### 13. 添加 Maven Wrapper

根目录缺 `mvnw` / `mvnw.cmd`，新成员需要自行安装 Maven。

```bash
mvn wrapper:wrapper
```

### 14. 补全 README 中的配置说明

**现状**：README 中的"基础设施"表格已全部替换为 `<your-server-ip>` 占位符。

已添加"第零步"说明了 `.template` → `.yml` 流程，但如果能从占位符中自动检测未替换的值并在启动时报错，会更好——例如启动时通过 `@PostConstruct` 校验 `${your_xxx}` 是否已被替换。

### 15. 启动顺序自动化

**现状**：需要手动按 3 阶段顺序启动 9 个服务。

**建议**：
- 提供一个 `start.sh` / `start.bat` 脚本，自动按依赖顺序启动各服务
- 各服务之间等待 Nacos 心跳确认后再启动下一个，而不是固定 sleep

---

## 总结

| 优先级 | 类别 | 项数 | 核心行动 |
|--------|------|------|---------|
| P0 | 安全 | 2 | 环境变量替换 + `.idea` 检查 |
| P1 | 工程基建 | 3 | 测试 + Docker + CI/CD |
| P2 | 代码/架构 | 7 | 错误码、日志、pom、Client、SQL、配置、分表 |
| P3 | 开发体验 | 3 | Maven Wrapper、启动脚本、校验 |

当前阶段建议优先投入 **P1**（尤其是 Docker 环境和测试体系），它们对团队协作和项目长期健康的边际收益最大。
