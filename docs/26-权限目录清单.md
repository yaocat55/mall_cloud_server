# 权限目录清单 — 接口与权限标识对照表

> 本文档供后端开发人员参考。每次新增接口时，对照此表确定对应的权限标识。
> 权限设计规范见 [25-RBAC设计规范-权限命名与设计方案.md](25-RBAC设计规范-权限命名与设计方案.md)。
>
> **更新日期：2026-07-10** — 所有 admin 路径已从 `/v1/xxx` → `/v1/auth/xxx`，新增 BFF 层路径映射。

---

## 架构说明

前端请求经过网关的完整路径链路：

```
前台：/api/admin/v1/{module}/{action}
        ↓（网关 stripPrefix=1，去掉 /api）
后台：/admin/v1/{module}/{action}          ← BFF 层（无权限校验，纯聚合）
        ↓（BFF 通过 Feign 转发）
服务：/v1/auth/{module}/{action}           ← 实际服务层（@PreAuthorize 在这里生效）
```

**列说明：**
| 列 | 说明 |
|----|------|
| BFF 路径 | 前端实际调用的接口路径 |
| 服务路径 | 后端服务实际处理的路径（@PreAuthorize 在此生效） |
| 权限标识 | Spring Security 的权限标记（`hasRole('admin')` 或 `hasAuthority('xxx')`） |

## 新增接口时检查

```markdown
- [ ] 这个接口属于哪个域（admin / basic / product / marketing / order）？
- [ ] 需要权限控制吗？
      → 不需要 → 加入 `PERMIT_ALL_URLS` 白名单并注明理由
      → 需要 → 继续往下填

- [ ] BFF 层是否已有对应的聚合接口？
      → 有 → 确认 BFF 路径和转发规则
      → 无 → 新增 BFF 端点或使用 ForwardController 透传

- [ ] 对应操作的权限标识：
      → admin 写操作 → `@PreAuthorize("hasRole('admin')")`
      → admin 读操作 → `@PreAuthorize("hasRole('admin')")`（同上）
      → C 端接口 → 不设 @PreAuthorize（通过 jwt 鉴权）

- [ ] auth_menu 表里有对应的记录吗？
      → 有 → 使用对应的 hasAuthority 标识
      → 无 → 等 seed SQL 更新后再上线

- [ ] 需要数据权限过滤吗？
      → 否 → 普通 @PreAuthorize
      → 是 → 额外改 BaseService 数据权限逻辑
```

> **注意：** 当前项目中所有 admin 写操作统一使用 `@PreAuthorize("hasRole('admin')")`，尚未按 menu 权限标识精确到单个操作。如需细粒度控制，参考 [24-RBAC设计规范.md](24-RBAC设计规范.md)。
