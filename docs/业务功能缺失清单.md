## P0 - 电商刚需缺失

### 1. 售后/退款流程

**现状：**
- `OrderReturnApplyService` 仅有一个 `applyStatus` 字段
- 无退款状态机流转
- 支付服务无数据库，退款无法记录

**需要补充的功能：**
- 退款单申请（用户 / 客服发起）
- 退款状态机：`WAIT_REFUND → REFUNDING → REFUNDED / CLOSED`
- 售后类型：仅退款（未发货）、退货退款（已收货）
- 退款原因分类（质量问题 / 7 天无理由 / 发错货 / 其他）
- 商家审核（同意 / 拒绝）
- 退货物流填写 + 商家收货确认
- 退款回写支付系统（调用支付宝退款接口）

**架构定位：退款编排放在 BFF 层**

退款涉及多个微服务协同，适合由 BFF 做编排（Saga 模式），而不是在一个业务服务内部跨服务调用：

```
用户/客服 → BFF（编排退款流程）
               ├─ ① mall-order：查询订单、创建退款单、更新订单退款状态
               ├─ ② mall-pay：调用支付宝退款 + 保存退款记录
               ├─ ③ mall-product：恢复已扣库存（已发货场景）
               └─ ④ mall-marketing：恢复已使用的优惠券
```

每个业务服务只需提供原子接口，由 BFF 控制流程和补偿。

两个 BFF 都参与：
- `mall-mobile-api`：用户手机端发起退款、查看进度、填写退货物流
- `mall-admin-api`：客服审核退款、确认收货、强制退款

**需要新增的表（`mall-order` 库）：**

```sql

    `refund_type`       tinyint      NOT NULL COMMENT '退款类型 1:仅退款 2:退货退款',
    `reason`            varchar(255) NOT NULL COMMENT '退款原因',
    `amount`            decimal(10,2) NOT NULL COMMENT '退款金额',
    `status`            tinyint      NOT NULL DEFAULT 0 COMMENT '状态 0:待审核 1:审核通过 2:已退款 3:已关闭 4:拒绝',
    `audit_time`        datetime              COMMENT '审核时间',

    `return_logistics_no`   varchar(128)       COMMENT '退货物流单号',
    `return_logistics_company` varchar(64)     COMMENT '退货物流公司',
    `confirm_time`      datetime              COMMENT '商家确认收货时间',
    `refund_time`       datetime              COMMENT '退款完成时间',
```

> **分片建议：** ✅ 必须分片。归属 `mall-order` 服务（数据与订单强关联），纳入 `mall_trade` 库群。  
> 分片键 `order_id`，算法 `order_id % 8`（库） / `order_id % 4`（表）——  
> 与现有 `t_order_return_apply` 一致。绑定到 `t_order` 绑定表组。

**涉及现有表变更：**
- `pay_order`（新建，见架构改进建议 P1-5）
- `order_order`（现有）：增加 `refund_status` 字段

### 2. 物流/配送体系

**现状：**
- 无物流模块
- 下单时不计算运费（运费为 0 或写死）
- 无发货操作

**需要补充的功能：**
- 运费模板（按件 / 按重量 / 按体积 / 满包邮）
- 配送区域配置（不同地区不同运费）
- 快递公司管理（顺丰/圆通/中通/韵达等）
- 订单发货（发货 → 填运单号 → 确认收货）
- 物流轨迹查询（对接快递鸟/快递100）

### 3. 商品搜索

**现状：**
- `mall-product` 依赖了 ES（`spring.elasticsearch` 有配置）
- 但无 ES 搜索接口，仅有 MySQL 的模糊查询

**需要补充的功能：**
- 商品全文搜索（名称/描述/关键词）
- 分类筛选 + 属性筛选聚合
- 排序（销量/价格/新品/评价）
- 搜索关键词联想
- 搜索历史记录
- ES 索引同步（RocketMQ 消费→增量同步 ES）

**需要新增的表：**

```sql
-- 搜索关键词记录
CREATE TABLE `product_search_keyword` (
    `id`                bigint       NOT NULL COMMENT '主键',
    `keyword`           varchar(128) NOT NULL COMMENT '搜索关键词',
    `search_count`      int          NOT NULL DEFAULT 1 COMMENT '搜索次数',
    `user_id`           bigint                COMMENT '用户ID（null=游客）',

    PRIMARY KEY (`id`),
    KEY `idx_keyword` (`keyword`),
    KEY `idx_user_id` (`user_id`)
) COMMENT '商品搜索关键词记录表';
```
> **分片建议：** ❌ 不分片。由 ES 承担全文搜索，MySQL 仅记录搜索热词做聚合展示，数据量可控。归属 `mall-product` 单库。

**涉及现有表变更：**
- ES 索引 `product_index`：映射 SPU + 价格区间 + 分类路径 + 标签
- 新增 ES 同步消费者：监听 RocketMQ 商品变更消息

### 4. 会员体系

**现状：**
- 移动端用户和 Admin 用户共用 `auth_user` 表
- 无会员等级、积分、成长值概念
- 详见 `docs/架构改进建议.md` P2-1

**需要补充的功能：**
- 会员等级（普通 / 银卡 / 金卡 / 钻石）
- 成长值（根据消费金额累积）
- 积分（消费赠送 + 积分抵扣）
- 会员权益（包邮、折扣价、专属优惠券、生日礼）
- 等级保级/降级规则
- 签到送积分

**需要新增的表：**

## P1 - 高频业务缺失

### 5. 促销活动（除优惠券外）

### 分片集群整合

```
现有分片库群：
  mall_trade_0~7     ← mall-order（订单、退款、发货、发票）
  mall_message_0~7   ← mall-message（通知、站内信）
  mall_recommend_0~7 ← mall-recommend（推荐、浏览记录）

新增分片库群：
  mall_member_0~7    ← mall-member（积分记录）

不分片库：
  mall-member         ← 会员基本信息、等级配置
  mall-product        ← 商品（搜索走 ES）、评价、收藏
  mall-marketing      ← 优惠券、促销活动、秒杀
  mall_basic          ← 基础服务
  mall_report         ← 统计报表
  mall_auth           ← RBAC 权限
```

**总计：新增约 6 张分片表、11 张单表 + ES 索引，改造约 6 张现有表。**
